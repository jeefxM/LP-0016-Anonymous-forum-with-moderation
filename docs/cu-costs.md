# Compute-unit (CU) cost of on-chain operations

Required by the bounty: the CU cost of each on-chain operation (registration,
slash submission) on LEZ. Two operations touch the chain; posting and
moderation are off-chain (ADR-001, docs/protocol.md).

> Note (from the bounty): LEZ's per-transaction compute budget may change
> during testnet. The numbers below are measured against a local standalone
> sequencer at LEZ rev `8c8f5b57` and should be re-measured on the public
> testnet when available.
>
> Official guidance (Logos team, #blockchain, 2026-05-12): **there are no
> official CUs for LEZ yet** â€” "you can measure cycles," and "we will not
> reject a L-Prize if there are no CUs, as CUs are not defined." The accepted

> below (plus the ADR-002 proving benchmark) therefore satisfy this

## How this was measured

A standalone LEZ sequencer was run with **`RISC0_DEV_MODE=0`** (confirmed in
the process environment), and the live-chain runners drove the full flow:

- `forum_register` â†’ `Initialize` + `Register`
- `forum_slash` â†’ `Initialize` + `Register` + `Slash`

Per-instruction execution cost is the risc0 **executor** time the sequencer
logs (`risc0_zkvm::...::executor: execution time`). End-to-end latency is wall
clock from the runner.

## Architecture note (important for interpreting "CU")

In this standalone configuration the sequencer **executes** each instruction
in the risc0 executor (the cost tabulated below) and settles to bedrock. It
does **not** generate the STARK proof inline â€” there are zero proving lines in
the sequencer log and execution is millisecond-scale. Real risc0 proof
generation is a separate step in this stack (the same way the feasibility
proof in ADR-002 was produced by the `proof-host` prover, not the sequencer).
So "CU" here has two layers: **execution compute** (below) and **proof
generation cost** (ADR-002 reference). Where the public testnet generates the
validity proof, and what it meters as the CU budget, is an open question for
the LEZ team.

## Execution compute (measured, `RISC0_DEV_MODE=0`)

| On-chain op | guest executor time | notes |
|---|---:|---|
| `Initialize` | ~6 ms | one PDA claim + state encode |
| `Register` | ~12â€“19 ms | Merkle fold (depth 16) + state encode |
| `Slash` | **~373 ms** | `verify_slash`: NĂ—K Ed25519 verifies + BN254 polynomial/Lagrange checks + Merkle fold |

Slash is ~20â€“60Ă— heavier than register, dominated by the in-zkVM Ed25519 and
BN254 work (ADR-008). This ratio is the load-bearing number: it's the relative
compute (and therefore the relative proving cost) of the two operations.

## End-to-end latency (measured)

| flow | wall clock | composition |
|---|---:|---|
| register (`forum_register`) | ~21 s | Initialize + Register |
| slash (`forum_slash`) | ~36 s | Initialize + Register + Slash |

Latency is **block-cadence-bound, not compute-bound**: the sequencer's
`block_create_timeout` is 15 s, so each instruction waits up to a block
interval. The compute itself is milliseconds.

## Proof generation cost (reference)

The representative cost of a real risc0 proof of an LEZ guest of this size,
measured with `proof-host` at `RISC0_DEV_MODE=0` (ADR-002): **2 segments /
524 288 cycles / ~55 s** on this CPU for the (now-legacy) post-proof guest.
`Register` is lighter (Merkle fold only â€” likely a single segment); `Slash` is
heavier (its ~373 ms execution implies materially more cycles â†’ more segments).
Exact per-instruction proof-cycle counts depend on where the validity proof is
generated on testnet (open question above) and are a follow-up measurement.
